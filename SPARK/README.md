# SPARK Standards

Copy these files into a SPARK/Ada project and run the tasks through `mise`.

This is a strict, systems-level generic starting template. It assumes new
SPARK-first code, not unconstrained Ada maintenance. Relax proof, runtime, or
toolchain checks only when a project has a clear target, compiler, or solver
reason.

## Tooling

```sh
mise run spark:install
mise run spark:standards
mise run spark:fmt:check
mise run spark:manifest
mise run spark:lint
mise run spark:prove
mise run spark:test
mise run spark:standards:check
```

The baseline pins Alire through mise, then pins GNAT, GPRbuild, GNATprove, and
GNATformat through `alire.toml`. `alire/alire.lock` is generated by Alire and
should be committed by downstream projects after the first successful
`spark:lock:check`.

The standards check formats Ada sources with GNATformat, rejects non-exact Alire
dependencies, compiles with GNAT warnings promoted to errors and runtime checks
enabled, proves all SPARK units with GNATprove level 4, treats proof warnings
and unproved checks as failures, and runs a small executable test project.

## Policy

- Keep production code in `src/`, tests in `tests/`, and generated build output
  under `build/` or `obj/`.
- Prefer SPARK package specs with explicit `Global`, `Depends`, `Pre`, and
  `Post` contracts on public subprograms.
- Keep `SPARK_Mode => On` as the default for reusable code. Isolate unavoidable
  full-Ada code behind narrow interfaces and document the proof boundary.
- Pin Alire dependencies with exact `=x.y.z` constraints. Do not use wildcard,
  range, or moving dependencies in the reusable baseline.
- Keep `mise run spark:prove` in the standards check, not as a separate
  release-only activity.

## Copy Notes

Rename `project_name` in `alire.toml`, project files, package names, and test
names before using this as a real project. If a project keeps different project
file names, set `SPARK_PROJECT_FILE` and `SPARK_TEST_PROJECT_FILE` in its mise
environment.
